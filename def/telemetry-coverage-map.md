## Purpose

This coverage map defines the **denominator** for Definition Manager control-flow telemetry coverage. Each row represents one **telemetry coverage unit** — a business-meaningful execution path or decision point that must be observable through logs, spans, or events.

A coverage unit is **satisfied** when test evidence or operational acceptance proves the expected telemetry exists with the required attributes. The Feature target (F-AC-8) is: **≥95% of `required` units must have passing evidence**, and **no critical state-changing flow** (Definition create/transform, Ascription lifecycle, bootstrap, error handling) may be uncovered regardless of aggregate percentage.

---

## Coverage Status Values

| Status                      | Meaning                                                                                   |
| --------------------------- | ----------------------------------------------------------------------------------------- |
| `required`                  | Must have test evidence; counted in coverage denominator                                  |
| `agent-only`                | OTel Java agent owns instrumentation; Defman documents but does not implement custom code |
| `no-instrumentation`        | Intentionally not instrumented (e.g., entity methods, internal helpers)                   |
| `dormant-until-code-exists` | Documented for future code; not counted until implemented                                 |

## Coverage Target (AC-5)

**Quantitative criterion:** ≥95% of `required` coverage units must have passing test evidence.

**Qualitative criterion:** No critical state-changing flow (Definition create/transform, Ascription lifecycle transitions, bootstrap seeding, caught error/warning handling) may be uncovered, regardless of aggregate percentage.

---

## Routing Table Verification (GAP-3)

Every flow in this map has been cross-checked against the Story routing table (S-004 implementation routing section). Each flow is assigned to exactly one implementing Story:

- **S-005:** HTTP entry, tenant extraction (present/missing), MDC cleanup
- **S-006:** JSON log rendering, log-trace correlation (in-span/no-span)
- **S-007:** Payload logging (under-cap/over-cap), sampled bypass
- **S-008:** Ascription lifecycle transitions, activation/deactivation hooks, cascade, cascade skip, rejected transition
- **S-009:** Definition create/transform, create/transform payload logging, subtype hooks, port derivation
- **S-010:** Definition read, repository/JDBC calls, entity no-instrumentation rule
- **S-011:** Caught unrecoverable error, caught recoverable warning, persistence exception translation, catch-site audit
- **S-012:** Mechanism Starlark parse/validation, Norm CEL validation, validation exception duality
- **S-013:** Bootstrap seed startup, per-kind events, seed failure, startup marker coherence
- **S-014:** Resource attributes, sampling config, log level config, collector outage resilience
- **S-015:** End-to-end trace shape, telemetry coverage calculation, F-AC matrix, perf baseline

No flow is assigned to multiple Stories. No flow is left unassigned (excluding `agent-only` and `no-instrumentation` which are documented but not implemented by Defman).

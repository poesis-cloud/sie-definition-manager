  note as CsdmNote
    This model defines a reference set of notions for defining systems, with the
    objective of being accepted as canonical:
    their components, capacities, interfaces, mechanisms, behaviors, and
    dynamics.

    It provides a structured framework for modeling complex systems of any type
    in a consistent and extensible manner, enabling **active** and **automatic**
    systemic design, cross-cutting causal analysis, and coherent end-to-end
    governance.

    In systemics theory terms, these notions and their relations are widely
    accepted as *second-order* elements: they define how to represent and define
    systems. That is why this model is said to be *systemic*.

    For IT profiles, a useful (but imperfect) analogy is Kubernetes' *control
    plane*: just like GSM, it defines governance constraints over a desired
    state, and it continuously acts to reach and maintain that desired state for
    the *data plane*. The Kubernetes *data plane*, like systems described with
    GSM, corresponds to *first-order* elements in systemics theory terms.

    The key conceptual difference is scope: Kubernetes targets one class of
    systems (container orchestration), whereas GSM is designed as a
    general-purpose definition model for systems of any kind.

    Even though GSM (the system definition informations structure) is
    general-purpose, it supports extensible information attributes and
    semantics. Structural constancy with semantic variation of informations
    is one key enabler of cross-domain, global systemic automatic reasoning.

    The autopoiesis (self-definition and self-maintenance) of systems is based
    on a systemic architecture pattern involving five universal organizational
    functions (GRS&CO) whose purpose is to define, enforce, and operate
    governance artifacts:

    - **Governor Mechanisms**: Define and enforce *principles*.
    - **Regulator Mechanisms**: Interpret *principles* and define
      *policies*, *rules*, and *controls*.
    - **Supervisor Mechanisms**: Enforce *policies*, *rules*, and
      *controls*.
    - **Coordinator Mechanisms**: Stabilize interactions between
      supervisors and operators (no artifact definition).
    - **Operator Mechanisms**: Operate *rules* and implement
      *controls* in the state plane.

    These function labels are **positional** (relative to system scope),
    not absolute: a Governor in one system may be an Operator from a
    supersystem's perspective (heterarchy).

    There are 4 universal governance artifacts (PPRC), each applying to a
    distinct systemic layer:

    - Principles: Identity-level, long-term normative constraints.
    - Policies: Design-level constraints that operationalize principles.
    - Rules: Logical constraints on behavior.
    - Controls: Monitoring, detection, correction, escalation mechanisms.

    This systemic synergy can be formulated as follows:

    Structure = Arch(Functional,Viability)
    Behavior  = Rule(Structure,Functional,Viability)

    This version 1.0.0 of the GSM focuses on core concepts and relationships,
    laying the foundation for future extensions and refinements.
  end note

  ' ============================================================

  class SubsystemIntegrationGovernance {
    +getName(): String
    +getSupersystem(): System
    +getSubsystem(): System
    +getLifecycleGovernance(): Boolean
    +getContractEnforcement(): Boolean
    +getConfigurationAuthority(): Boolean
    +getSubstitutionAuthority(): Boolean
    +getViabilityAccountability(): Boolean
    +getResourceAllocation(): Boolean
    +getBoundaryInclusion(): Boolean
    +getPurposeContribution(): Purpose[]
  }

  note top of SubsystemIntegrationGovernance
    Captures the explicit governance relationship between a supersystem and a
    subsystem. "Subsystem" is not an intrinsic property—it is a relational role
    defined by governable integration, not by internal control.

    A system S is a subsystem of supersystem P if P can govern the *integration
    relationship* with S — not necessarily S's internals.

    **Systemic Semantics**:
    - A SubsystemIntegrationGovernance makes explicit *why* a system is
      considered a subsystem of another, capturing which governance levers the
      supersystem holds over the relationship.
    - Heterarchy is natural: a system may be a subsystem of multiple
      supersystems via distinct SubsystemIntegrationGovernance instances, each
      governing different aspects.
    - The subsystem retains its own identity, boundary, and potentially its own
      governance — what makes it "sub" is the supersystem's authority over the
      integration relationship, not over subsystem internals.

    **Attribute Semantics**:
    - "name": Human-readable semantic identifier for the integration.
    - "supersystem": The system that governs the integration relationship, the
      owner of this primitive instance.
    - "subsystem": The system whose integration is governed.
    - "lifecycleGovernance": Supersystem controls provisioning, versioning,
      upgrade cadence, or decommissioning of the subsystem.
    - "contractEnforcement": Supersystem can enforce interaction contracts
      (interfaces, SLAs, event schemas) on the subsystem.
    - "configurationAuthority": Supersystem can restrict or mandate subsystem
      configuration (modes, features, parameters).
    - "substitutionAuthority": Supersystem can replace the subsystem with an
      alternative implementation.
    - "viabilityAccountability": Supersystem is accountable for subsystem
      viability (incidents in sub are incidents of super).
    - "resourceAllocation": Supersystem controls resource budgets (compute,
      memory, rate limits) allocated to the subsystem.
    - "boundaryInclusion": Subsystem boundary is operationally inside
      supersystem boundary.
    - "purposeContribution": Purposes of the supersystem that the subsystem
      materially contributes to.

    **Derivation Rules**:
    - If ALL governance booleans are false AND purposeContribution is empty,
      the relation SHOULD be reconsidered: S may be an external dependency
      rather than a subsystem.

    **Examples** (Software Application as supersystem):

    1. *OS as subsystem*:
       - substitutionAuthority: true (app decides which OS to support)
       - contractEnforcement: true (app defines required OS API contract)
       - configurationAuthority: true (app may mandate OS settings)
       - viabilityAccountability: true (OS failure = app failure)
       - purposeContribution: [Execution Platform]

    2. *VM/Container Runtime as subsystem*:
       - resourceAllocation: true (app defines resource requests/limits)
       - boundaryInclusion: true (VM hosts the app boundary)
       - substitutionAuthority: true (app can target different runtimes)
       - viabilityAccountability: true (runtime failure = app failure)
       - purposeContribution: [Execution Platform]

    3. *IT RUN Team as subsystem*:
       - lifecycleGovernance: true (app defines release/deploy cadence)
       - contractEnforcement: true (app defines SLAs, runbooks)
       - viabilityAccountability: true (ops incidents = app incidents)
       - purposeContribution: [Operational Continuity]

    4. *Library as subsystem*:
       - lifecycleGovernance: true (app controls version upgrades)
       - substitutionAuthority: true (app can swap implementations)
       - contractEnforcement: true (app defines expected API)
       - boundaryInclusion: true (library runs inside app boundary)
       - purposeContribution: [specific functional purposes]

    These examples illustrate that "subsystem" is defined by *who governs the
    integration*, not by relative size or abstraction layer.

    **Consistency Rules**:
    - "name" MUST be maximum 80 characters long.
    - "supersystem" MUST be defined.
    - "subsystem" MUST be defined.
    - "supersystem" MUST NOT be the same as "subsystem".
    - At least one governance lever (lifecycleGovernance, contractEnforcement,
      configurationAuthority, substitutionAuthority, viabilityAccountability,
      resourceAllocation, boundaryInclusion) MUST be true, OR
      purposeContribution MUST NOT be empty.
    - A system MUST NOT have a SubsystemIntegrationGovernance where it is both
      supersystem and subsystem (no direct self-loops).
    - Cycles in SubsystemIntegrationGovernance chains are allowed (heterarchy),
      but SHOULD be flagged for review if they form tight loops.
  end note

  abstract class Policy {
    +getName(): String
    +getStatus(): PolicyStatus
    +getApplicabilityConditions(): String
    +getConstraint(): String
    +getPrinciples(): Principle[]
  }

  enum PolicyStatus {
    DRAFT
    PROPOSED
    APPROVED
    REJECTED
    PUBLISHED
    INTEGRATED
    SUSPENDED
    DEPRECATED
  }

  note top of Policy
    A policy is a measurable normative regulatory constraint over a system
    definition. It does not prescribe behavior directly; instead, it constrains
    what the system MUST do (functional), MUST withstand (viability), or MUST be
    (structural).

    If a constraint is not measurable, it is not a policy.

    **Systemic Semantics**:
    - Policies are DEFINED by Regulator mechanisms.
    - Policies are ENFORCED and OPERATED by Supervisor mechanisms.
    - Policies operationalize one or more Principles.

    **Homeostatic Auto-Definition Rule (Sensor)**:
    When a Policy constraint is measurable, the system SHOULD auto-define:
    - A **Sensor** that measures the criteria
    - An **EventArchetype** for the measurement (MeasurementEvent)
    - An **Interaction** from Sensor.Effector → Governor.Receptor
    This closes the homeostatic loop: Governor defines Policy → Sensor
    measures → Governor receives feedback → Governor adjusts Policy.

    **Multi-Dimensional Application**:
    - META (GSM): Defines the pattern (how governance SHOULD work)
    - OPERATIONAL (SIE): SIE's processes realize this pattern
      (Kafka consumers = Sensors, CoherenceCheck = Governor feedback)
    - CONTROL (Actuators): Execute actions on State Plane
    - STATE (External): Actual systems emit metrics consumed by SIE

    **Attribute Semantics**:
    - "name": Human-readable semantic identifier.
    - "applicability conditions": A logical expression that must evaluate to
      true for the policy to be applicable to the system.
    - "constraint": A declarative expression of the constraint imposed by the
      policy.

    **Evaluation Semantics (derived)**:
    Policy evaluation is derived deterministically from the constraint format.
    The policy definition does not carry an additional author-defined
    evaluation field.

    **Rationale (avoid duplicated semantics):**
    Some constraint formats (notably **ThresholdConstraint**) already embed the
    key measurement semantics required for evaluation (metric + aggregation +
    timeWindow). Duplicating measurement semantics in a separate field invites
    divergence and non-reproducible evaluation.

    **Validation Rules**:
    - *applicability conditions* MUST be a valid logical operator GSM-DSL JSON AST
      node.
    - *constraint* MUST be a valid constraint format GSM-DSL JSON AST node.

    **Guard × Constraint Compatibility Rules**:
    A Policy is only well-defined if its applicability guard and constraint can
    be evaluated together.

    - **Definedness:** For every evaluation context $C$ where
      `applicabilityConditions(C) == true`, the `constraint(C)` evaluation MUST
      be defined (no missing references, null-dereferences, invalid paths, etc.).
    - **Type soundness:** Under the same contexts, the constraint MUST be
      type-consistent (e.g., do not compare `duration` to `number`, do not apply
      numeric aggregation to a `string`-typed attribute).
    - **Domain alignment (selectors):** If the constraint quantifies over or
      selects StateObjects (explicit selector in the constraint format), then
      the applicability guard MUST NOT allow contexts where that selector’s
      required archetype(s) cannot exist.
      (Otherwise the constraint becomes vacuous/undefined depending on the
      operator semantics.)
    - **Optional (quality):** The system MAY warn when `applicabilityConditions`
      is satisfiable but makes the constraint always trivially true/false.

    **Constraint → Evaluation Derivation Rules (GSM-DSL v1):**
    The following derivations are mandatory and deterministic:

    1) ThresholdConstraint
       - Measure the referenced metric using the declared aggregation and
         timeWindow, then compare to the threshold.

    2) InvariantConstraint
       - Evaluate the invariant boolean condition against the relevant state
         objects.

    3) DependencyConstraint
       - Evaluate the structural dependency relation over the state graph
         implied by the selected state objects.

    4) CardinalityConstraint
       - Count matching state objects and evaluate the bounds.

    **Consistency Rules**:
    - "name" MUST be maximum 80 characters long.
    - "name" MUST be unique among all policies of the same system.
    - A policy MUST have at least one principle.
  end note

  class Rule {

  note top of Rule
    A behavioral rule is a constraint over the system's behavioral properties:
    how the system acts over time via mechanisms, sensors, and effectors.

    **Systemic Semantics**:
    - Behavioral rules operationalize policies:
      "applyingPolicies" references the functional, viability, and/or
      structural policies whose constraints this rule enforces.
    - Rules are DEFINED by Regulator controller mechanisms.
    - Rules are ENFORCED by Supervisor controller mechanisms via Controls.
    - Rules are OPERATED by Operator controller mechanisms.

    **Attribute Semantics**:
    - "name": Human-readable semantic identifier.
    - "applying functional policies": The set of functional policies whose
      constraints are enforced by the behavioral rule.
    - "applying viability policies": The set of viability policies whose
      constraints are enforced by the behavioral rule.
    - "applying structural policies": The set of structural policies whose
      constraints are enforced by the behavioral rule.
    - "scoped functions": The set of functions whose execution the
      behavioral rule may impact when applied.
    - "scoped modalities": The set of modalities whose state the behavioral
      rule may impact when applied.
    - "triggering receptor": The receptor whose received events trigger the
      behavioral rule.
    - "applicability conditions": A logical expression that must evaluate to
      true for the behavioral rule to be applied / operate behavioral rule
      effects.
    - "iteration conditions": A logical expression that, if true, causes the
      controller to re-apply the behavioral rule.
    - "control sequence": An ordered list of behavioral rule controls defining
      effectors when the behavioral rule is applied.

    **Validation Rules**:
    - *applicability conditions* MUST be a valid logical operator GSM-DSL JSON AST
      node.
    - *iteration conditions* MUST be a valid logical operator GSM-DSL JSON AST
      node.
    - The conditions operands referencing event properties MUST exist in the
      corresponding event archetype schemas beared by the mechanism receptor.
    - The conditions operands representing a literal value MUST be validated
      against the corresponding event archetype schema beared by the mechanism
      receptor.

    **Consistency Rules**:
    - A behavioral rule MUST be triggered by exactly one event archetype.
    - A behavioral rule MUST have at least one behavioral rule effect.
    - If a behavioral rule is deleted, its behavioral rule effects MUST be
      deleted, and vice-versa IF it is the last behavioral rule effect the
      behavioral rule owns.
    - MUST reference at least one policy in "applyingPolicies".
    - "scope" MUST be compatible with the policies' scope.
    - "trigger" MUST reference an event archetype defined in the same system.

    **Todo**:
    - Implement join/fork of effect sequences (which may define parallel
      execution).
  end note

  class Control {

  note top of Control
    A behavioral rule control defines an action to be performed when its
    enclosing behavioral rule is triggered and applied.

    **Systemic Semantics**:
    - Controls are DEFINED by Regulator controller mechanisms.
    - Controls are ENFORCED by Supervisor controller mechanisms.
    - Controls are OPERATED by Supervisor (activation) and Operator (implementation).

    **Attribute Semantics**:
    - "name": Human-readable semantic identifier.
    - "applying rule": The enclosing behavioral rule to which the behavioral
      rule control belongs.
    - "applicability conditions": A logical expression that must evaluate to
      true for the behavioral rule control to be applied when the enclosing
      behavioral rule is applied.
    - "iteration conditions": A logical expression that, if true, causes the
      controller to re-execute and re-evaluate the behavioral rule control.
    - "effector": The effector to be activated when the behavioral rule control
      is applicated.
    - "effector event source mapping": The mapping defining how the produced
      event instance is constructed from source events and scalars.
    - "feedback receptor": The receptor to receive feedback from the effect.
    - "actuation": Optional actuation explicitly declared by the control,
      triggering auto-definition of actuator/sensor mechanisms.

    **Auto Definition Rules**:
    - When a behavioral rule control is added to a behavioral rule, required
      source event archetypes SHOULD be inferred from the selected effector's
      mapping.
    - If the enclosing mechanism does not yet have receptors for those source
      archetypes, the application SHOULD propose creating them.
    - If the required source archetype does not match the behavioral rule's
      triggering event, the application SHOULD propose creating a new behavioral
      rule control behavioral rule triggered by that source archetype (chained
      rules).
    - Behavioral rule control conditions SHOULD apply rule definitions owned by
      supervisors. Hence, a behavioral rule control SHOULD be associated to at
      least one rule definition, indirectly, via interactions of the controller
      owning the conditions and supervisors owning the related rules. For such
      interaction to operate, the controller mechanism MUST have receptors
      receiving the Supervision event archetypes produced by the supervisors'
      effectors.
    - When a behavioral rule control is added to the system, the application
      SHOULD propose associating it to relevant rule definitions owned with
      supervisors.
    - When a rule definition is added to the system, the application SHOULD
      propose adding corresponding conditions to relevant behavioral rule
      controls.
    - When a behavioral rule control declares an Actuation, the system MUST
      auto-define:
      - An Actuator composed by that Actuation, with:
        - A receptor bound to the control effector's event archetype.
        - An effector bound to an auto-defined ActuationEvent archetype.
      - A Sensing (and Sensor) only if a feedback receptor is defined
        on the control, with:
        - A receptor bound to the actuator effector's event archetype.
        - An effector bound to the feedback receptor's event archetype.

    **Validation Rules**:
    - *applicability conditions* MUST be a valid logical operator GSM-DSL JSON AST
      node.
    - *iteration conditions* MUST be a valid logical operator GSM-DSL JSON AST
      node.
    - The conditions operands referencing event properties MUST exist in the
      corresponding event archetype schemas beared by the mechanism receptor.
    - The conditions operands representing a literal value MUST be validated
      against the corresponding event archetype schema beared by the mechanism
      receptor.

    **Consistency Rules**:
    - "name" MUST be maximum 80 characters long.
    - "name" MUST be unique among all behavioral rule controls of the system.
    - A behavioral rule control MUST have exactly one effector.
    - If a behavioral rule control is deleted, its effector MUST be deleted, and
      vice-versa.
    - A behavioral rule control SHOULD be associated to zero or more supervision
      rule definitions.
    - An behavioral rule control MUST have exactly one event mapping defining
      how the produced event instance is constructed from source receptor's
      events.
    - If the behavioral rule control is deleted, its event mapping MUST be
      deleted.
    - The event mapping MUST NOT be deleted without its behavioral rule control
      being deleted.
  end note

  ' SOURCE MAPPING
  ' ============================================================

  class SourceMapping {
    +getValueMapping(): String
  }

  note top of SourceMapping
    Captures, at design time, how an effector's produced event instance is
    constructed from one or more source events and scalars.

    This enables end-to-end data transformation traceability.

    The mapping is represented as a declarative value mapping specification.
    It is intentionally *not* decomposed as one object per target property to
    keep the conceptual model compact while still supporting property-level
    lineage.

    Example structure (illustrative):
    - keys: target property paths
    - values: source event property paths OR JSON literals

    A SourceEventPropertyPath includes the reference to the source event
    archetype it is defined on.

    **Event Mapping Rules**
    - For each effector event source mapping M:
      - Ensure M.targetEventArchetype == effector.on (target event archetype).
    - Ensure a receptor exists for each distinct source event archetype
      referenced by any SourceEventPropertyPath in M.valueMapping.
    - Ensure a rule exists for each such distinct source event archetype,
      with triggering event == that source event archetype.
    - Ensure a rule effect exists in that rule which selects the effector.
    - When multiple rule effects exist in a rule, order deterministically
      (e.g. by effector name) to keep regeneration idempotent.

    **Consistency Rules**:
    - A mapping MUST reference exactly one target event archetype (the one
      bound to the effector).
    - A mapping MAY reference zero or more source event archetypes via
      SourceEventPropertyPath values (and may be literals-only).
    - Each valueMapping key MUST be a TargetEventPropertyPath valid with respect
      to M.targetEventArchetype's schema.
    - Each valueMapping value MUST be either:
      - a SourceEventPropertyPath valid with respect to the schema of its
        referenced source event archetype, OR
      - a JsonLiteral value.
    - For each (targetPath -> value) entry, the value type MUST be compatible
      with the type of the target property specified by the target event
      archetype schema at targetPath:
      - If value is a SourceEventPropertyPath, the referenced source property
        type MUST match (or be safely coercible to) the target property type.
      - If value is a JsonLiteral, the JSON value MUST validate against the
        target property type.
  end note

    - "modalities": Modalities supported/managed by mechanisms in the
      component.
    - "mechanisms": Mechanisms belonging to the component.
    - "interfaces": Interfaces regrouped by the component (may be internal or
      external).
    - "interactions": Interactions in which the component participates.

    **Derivation Rules**:
    - "interfaces" MAY be computed from mechanisms grouped by the component.
    - "interactions" MAY be computed from:
      - interactions carried by channels of those interfaces, and/or
      - interactions whose effector/receptor is owned by those mechanisms.

    **Consistency Rules**:
    - "name" MUST be maximum 80 characters long.
    - "name" MUST be unique among all components of the same system.
  end note

  Component "1" o--> "1..*" Function : regroups

  ' INTERFACE
  ' ============================================================

  class Interface {
    +getName(): String
    +getReceptors(): Receptor[]
    +getEffectors(): Effector[]
    +getChannels(): Channel[]
  }

  note top of Interface
    An interface is an interaction surface of a system or component.

    Interfaces can be purely internal (structuring internal architecture) or
    external (participating in a system boundary).

    **Systemic Semantics**:
    - Interfaces group receptors and effectors to make interaction points
      explicit.
    - Channels are structural carriers that connect interfaces.

    **Attribute Semantics**:
    - "name": Human-readable semantic identifier.
    - "receptors": Receptors regrouped by the interface.
    - "effectors": Effectors regrouped by the interface.
    - "channels": Channels for which this interface is an endpoint.

    **Derivation Rules** (non-normative):
    - "receptors" and "effectors" MAY be computed from the interactions carried
      by the interface's channels (collecting interaction.receptor and
      interaction.effector, respectively).
    - "channels" MAY be computed from boundary/external connectivity design.
    - "channels" MAY also be computed as all channels for which this interface
      is an endpoint.

    **Consistency Rules**:
    - "name" MUST be maximum 80 characters long.
    - "name" MUST be unique among all interfaces of the same system.
    - An interface MUST regroup at least one receptor.
    - An interface MUST regroup at least one effector.
  end note

  SystemicPrimitive <|-- Interface : is
  Interface "1" o--> "1..*" Receptor : regroups
  Interface "1" o--> "1..*" Effector : regroups
  Interface "1" *--> "0..*" Channel : has

  ' CHANNEL
  ' ============================================================

  class Channel {
    +getName(): String
    +getProtocol(): String
    +getInteractions(): Interaction[]
    +getInterfaceA(): Interface
    +getInterfaceB(): Interface
  }

  note top of Channel
    A channel is a structural carrier that mediates interactions.

    A channel connects exactly two interfaces (interface A and interface B).

    Endpoints are intentionally unordered at this abstraction level; any
    directionality (client/server, producer/consumer, origin/destination) is a
    property of specific interactions carried by the channel.

    **Systemic Semantics**:
    - Channels model *how* exchanges are carried (transport/medium), not their
      meaning.
    - Multiple interactions may be carried by the same channel.

    **Attribute Semantics**:
    - "name": Human-readable semantic identifier.
    - "protocol": The exchange protocol/medium (e.g., HTTP, Kafka, in-memory).
    - "interactions": Interactions carried by the channel.
    - "interface A": One endpoint interface of the channel.
    - "interface B": The other endpoint interface of the channel.

    **Consistency Rules**:
    - "name" MUST be maximum 80 characters long.
    - "name" MUST be unique among all channels of the same system.
    - "interface A" MUST be defined.
    - "interface B" MUST be defined.
    - "interface A" MUST NOT be the same as "interface B".
  end note

  SystemicPrimitive <|-- Channel : is
  Channel "1" o--> "0..*" Interaction : carries
  Channel "1" o--> "1" Interface : interfaceA
  Channel "1" o--> "1" Interface : interfaceB

  class Interaction {
    +getName(): String
    +getEffector(): Effector
    +getReceptor(): Receptor

  note top of Interaction
    An interaction is an event exchange relationship between mechanisms.

    Interactions may be internal to a system (no channel, no external
    interface) or external (carried by channels bound to interfaces).

    **Systemic Semantics**:
    - An interaction is an atomic causal coupling from a producing effector to
      a receiving receptor.
    - The interaction's meaning is about *what* is exchanged, not the transport
      mechanism; transport may be modeled via channels.

    **Attribute Semantics**:
    - "name": Human-readable semantic identifier.
    - "effector": The producing endpoint emitting an event instance.
    - "receptor": The receiving endpoint consuming an event instance.

    **Consistency Rules**:
    - "name" MUST be maximum 80 characters long.
    - "name" MUST be unique among all interactions of the same system.
    - "effector" MUST be defined.
    - "receptor" MUST be defined.
    - The event archetype produced by "effector" MUST be compatible with the
      event archetype received by "receptor".
  end note

  Interaction "1" o--> "1" Effector : has

  ' MECHANISM
  ' ============================================================

  abstract class Mechanism {
    +getName(): String
    +getReceptors(): Receptor[]
    +getEffectors(): Effector[]
  }

  note top of Mechanism
    A mechanism represents a system's capability to process input events
    received via its receptors, and produce output events via its effectors.

    A mechanism represents an atomic causal unit of behavior within a system.

    **Positional semantics (GRS&CO):**
    The five mechanism subtypes (Governor, Regulator, Supervisor, Coordinator,
    Operator) are **positional labels**, not absolute roles. A mechanism's label
    is determined by its function relative to the system scope it governs.
    A Governor in one system may be an Operator from a supersystem's perspective
    (heterarchy; second-order observer dependence — von Foerster).
    This is consistent with SubsystemIntegrationGovernance.

    **Layer and plane placement:**
    - Governor belongs to the **Governance plane** (Reflexive Layer).
    - Regulator, Supervisor belong to the **Regulation plane** (Reflexive Layer).
    - Coordinator belongs to the **Coordination plane** (Reflexive Layer).
    - Operator belongs to the **Operation plane** (Ontic Layer).

    **Consistency Rules**:
    - "name" MUST be maximum 80 characters long.
    - "name" MUST be unique among all mechanisms of the same system.
    - A mechanism MUST have at least one receptor and one effector.
    - If a mechanism is deleted, its receptors and effectors MUST be
      deleted.
    - If a receptor or effector is deleted, the mechanism MUST NOT be deleted,
      except if the receptor or effector is the last one the mechanism owns.
  end note

  SystemicPrimitive <|-- Mechanism : is
  Mechanism "1" *--> "1..*" Receptor : has
  Mechanism "1" *--> "1..*" Effector : has

  ' RECEPTOR
  ' ============================================================

  class Receptor {
    +getName(): String
    +getReceivedEventArchetype(): EventArchetype
  }

  note top of Receptor
    A receptor represents a mechanism's capability to receive events produced by
    mechanism's effectors. Such received event represent a cause for the
    mechanism to operate.

    **Systemic Semantics**:
    Multiple receptors can listen to the same EventArchetype. This enables:
    - Multiple mechanisms to react to the same event type
    - Fan-out patterns where one event triggers many behaviors
    - Cross-component coordination via shared event contracts

    **Consistency Rules**:
    - "name" MUST be maximum 80 characters long.
    - "name" MUST be unique among all receptors of the same system.
    - A receptor MUST be bound to exactly one event archetype.
    - An event archetype MAY be bound to multiple receptors (sharing allowed).
    - If the bound event archetype is deleted, all receptors bound to it MUST
      be deleted.
    - If a receptor is deleted, the bound event archetype MUST NOT be deleted
      (other receptors may still reference it).
  end note

  SystemicPrimitive <|-- Receptor : is
  Receptor "0..*" o--> "1" EventArchetype : receives

    - If an effector is deleted, the bound event archetype MUST NOT be deleted

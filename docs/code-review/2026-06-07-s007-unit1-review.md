# Code Review: S-007 Unit 1 (PayloadLogHelper)

**Ready for Production**: No
**Critical Issues**: 1

## Priority 1 (Must Fix) ⛔

- Non-positive cap bypasses truncation and returns full payload (`capBytes <= 0` returns input unchanged), which can leak sensitive uncapped content if configuration is invalid or overridden at runtime.
  - Location: `src/main/java/cloud/poesis/sie/defman/observability/PayloadLogHelper.java`
  - Fix: For non-null payload and `capBytes <= 0`, return a safe marker-only value (possibly truncated) instead of raw payload.

## Priority 2 (Should Fix)

- Marker integrity is not guaranteed when cap is smaller than marker length: current branch truncates marker text itself, which can remove structured suffix semantics.

  - Location: `src/main/java/cloud/poesis/sie/defman/observability/PayloadLogHelper.java`
  - Fix: Keep marker parseable and deterministic under tiny caps (for example, reserve a minimal fallback marker and truncate preview only).

- Tests do not cover high-risk boundary conditions:
  - `capBytes <= 0` must not return raw payload.
  - `capBytes < markerBytes` should preserve deterministic marker semantics.
  - Surrogate pair boundary behavior for UTF-8 multi-byte code points.
  - Location: `src/test/java/cloud/poesis/sie/defman/observability/PayloadLogHelperTest.java`

## Recommended Minimal Patch

1. In `PayloadLogHelper.boundedPayload(...)`, change non-positive cap handling from returning raw payload to returning bounded marker output.
2. Introduce a tiny-cap fallback marker strategy so output remains explicit and bounded.
3. Add tests for non-positive cap, tiny cap, and surrogate pair truncation safety.
